# CLAUDE.md

이 문서는 본 프로젝트에서 Claude(및 모든 기여자)가 따라야 할 **개발 규칙**과 **디자인 규칙**을 정의한다.
계획·일정·단계는 `plan.md`를 참고할 것. 이 문서는 "어떻게 만드는가"의 규칙만 다룬다.

---

## 프로젝트 한 줄 요약

설치 없이 브라우저만으로 음원 녹음 · 리믹스 · 드럼 연주 · 클럽 디제잉을 할 수 있는 무료 웹 앱.
모든 오디오 처리는 클라이언트에서 수행하며, 작업물은 서버 저장 없이 사용자가 직접 파일로 내려받는다.

---

## 개발 규칙

### 기술 스택 (고정)

- **프레임워크**: next.js + TypeScript
- **오디오 엔진**: Tone.js (신스 · 시퀀서 · 이펙트 · Transport · 샘플 재생)
- **저수준 오디오**: Web Audio API (GainNode, BiquadFilter, PannerNode, MediaStreamDestination 등)
- **녹음**: `getUserMedia` + `MediaRecorder`
- **MP3 인코딩**: lamejs (브라우저 내 WAV → MP3 변환)
- **파형 시각화**: Wavesurfer.js
- **스타일**: Tailwind CSS
- **배포**: Netlify 또는 Vercel (정적 배포)

스택을 바꾸거나 새 라이브러리를 추가할 때는 반드시 이유를 PR 설명에 적는다. "있으면 편할 것 같아서"는 사유가 아니다. 번들 크기에 직접 영향을 주는 오디오/UI 라이브러리는 특히 보수적으로 판단한다.

### 아키텍처 원칙

- **클라이언트 전용.** 백엔드·DB·서버 저장은 도입하지 않는다. 사용자의 녹음물·업로드 파일은 절대 외부로 전송하지 않는다.
- **오디오 엔진은 UI와 분리한다.** Tone.js / Web Audio 로직은 별도 모듈(예: `src/audio/`)에 두고, React 컴포넌트는 이를 호출만 한다. 컴포넌트 안에 오디오 노드 생성 로직을 직접 박지 않는다.
- **단일 AudioContext.** 앱 전체에서 AudioContext는 하나만 생성해 공유한다. 섹션마다 새로 만들지 않는다.
- **섹션 간 독립성.** 녹음 · 리믹스 · 드럼 · DJ 각 섹션은 서로의 내부 구현에 의존하지 않는다. 공유가 필요하면 공통 오디오 모듈을 통해서만 한다.

### 오디오 필수 규칙 (어기면 소리가 안 난다)

- **AudioContext는 사용자 제스처 이후에만 시작한다.** 첫 클릭/터치에서 `Tone.start()` 또는 `audioContext.resume()`를 호출하는 언락 게이트를 반드시 둔다. 페이지 로드 시점에 자동 시작 금지.
- **iOS Safari 언락을 별도 처리한다.** 첫 사용자 제스처에서 무음 버퍼를 한 번 재생해 오디오를 깨운다. iOS는 지연(latency)이 크므로 라이브 연주 기능은 실제 기기에서 검증한다.
- **마이크 접근은 HTTPS 또는 localhost에서만 동작한다.** 배포본(Netlify/Vercel 자동 HTTPS)은 문제없으나, 권한 거부 상태를 항상 우아하게 처리한다.
- **오디오 노드는 반드시 정리한다.** 컴포넌트 언마운트 / 섹션 전환 시 `stop()`, `disconnect()`, 스트림 트랙 `stop()`을 호출한다. 노드 누수는 시간이 지나면 끊김·크래시로 이어진다.
- **무거운 변환(MP3 인코딩 등)은 메인 스레드를 막지 않게 한다.** 가능하면 Web Worker로 분리하고, 진행 표시를 제공한다.

### 권한 · 프라이버시

- 마이크 권한은 사용자가 녹음을 시도하는 시점에만 요청한다. 앱 진입 시 선요청 금지.
- 권한 거부 · 미지원 브라우저 상황에 대한 안내 UI를 항상 갖춘다.
- 외부 음원은 사용자 업로드와 명시적으로 라이선스가 확인된 무료 음원만 사용한다. YouTube 등에서의 오디오 추출은 기술적·법적 이유로 절대 구현하지 않는다.

### 코드 컨벤션

- TypeScript `strict` 모드 사용. `any`는 원칙적으로 금지하고 불가피하면 주석으로 사유를 남긴다.
- 함수 컴포넌트 + Hooks만 사용. 클래스 컴포넌트 금지.
- 파일/폴더는 기능 단위(feature-based)로 묶는다. 예: `src/sections/dj/`, `src/sections/drum/`.
- 매직 넘버(BPM 기본값, 스텝 수, 패드 수 등)는 상수로 분리한다.
- 커밋 메시지는 명령형 한 줄 요약 + 필요시 본문. 한 커밋은 한 가지 일만 한다.
- 외부 라이브러리 API는 직접 호출하기보다 얇은 래퍼를 거치게 해 교체 가능성을 남긴다.

### 테스트 / 검증

- 새 기능은 **데스크탑 Chrome + 모바일 Safari(iOS)** 최소 두 환경에서 직접 소리를 확인한 뒤 머지한다.
- 오디오는 자동 테스트가 어려우므로, 수동 검증 체크리스트(소리 남 / 끊김 없음 / 정리됨 / 권한 거부 처리됨)를 PR에 적는다.

## 디자인 규칙

### 기본 방향

- **다크 테마 기본.** 음악 제작 도구의 관례이자 장시간 작업 시 눈 피로가 적다. 라이트 테마는 후순위.
- **도구처럼 보이게, 장난감처럼 느껴지지 않게.** 과한 그라데이션·이모지 남발 대신 명확한 컨트롤과 위계.
- **즉각적 피드백.** 모든 오디오 동작(재생·녹음·패드 타격)은 시각적으로 즉시 반응한다. 무음의 클릭은 금지.

### 반응형 · 터치

- **모바일 퍼스트로 설계하되 데스크탑에서 공간을 활용한다.** 좁은 화면에서 먼저 동작하게 만들고 넓어지면 확장.
- **터치 타깃 최소 44×44px.** 드럼 패드·DJ 컨트롤 등 연주용 요소는 더 크게.
- 드럼 패드, 크로스페이더, 노브 등은 손가락으로 정확히 칠 수 있는 크기와 간격을 확보한다.
- 가로/세로 회전 모두 깨지지 않게 한다. DJ·믹서 화면은 가로 모드를 권장 안내해도 좋다.
- 스크롤과 제스처 충돌 주의: 패드 영역·페이더 드래그에서 페이지 스크롤이 끼어들지 않게 처리한다.

### 레이아웃

- 상단 또는 하단에 4개 섹션(녹음 / 리믹스 / 드럼 / DJ) 네비게이션을 둔다. 모바일은 하단 탭바 권장.
- 각 섹션은 독립된 화면처럼 동작하되, 공통 헤더(전송 컨트롤·언락 상태·BPM 등 전역 상태)는 일관되게 유지한다.
- 정보 위계: 지금 조작할 컨트롤 > 상태 표시 > 보조 기능. 핵심 컨트롤이 한눈에 들어와야 한다.

### 컬러 · 타이포그래피

- 어두운 중성 배경(차콜/거의 검정) + 1~2개의 강조색(녹음=레드 계열 등 기능적 의미 부여).
- 강조색은 "지금 활성/녹음 중" 같은 상태 신호로 아껴 쓴다. 장식용 남발 금지.
- 숫자/시간/BPM 등 수치는 고정폭(monospace) 또는 tabular-nums로 흔들리지 않게.
- 본문은 가독성 높은 산세리프. 폰트 종류는 최소화(2종 이내).

### 상태 표현 (오디오 앱 특화)

- **녹음 중**은 누구도 헷갈리지 않게 강하게 표시한다(레드 점멸 + 경과 시간).
- 재생/정지, 활성 트랙, 뮤트/솔로, 로딩(샘플·인코딩) 상태를 명확한 아이콘·색으로 구분한다.
- 권한 거부, 브라우저 미지원, 오디오 미언락 상태는 차단 화면이 아니라 해결 방법을 안내하는 메시지로 보여준다.

### 접근성

- 색만으로 상태를 전달하지 않는다(아이콘·텍스트 병행).
- 키보드로도 주요 동작(재생/정지/녹음)이 가능하게 한다.
- 명도 대비는 WCAG AA 이상을 목표로 한다.

## 하지 않을 것 (Non-goals)

- 백엔드 / 사용자 계정 / 클라우드 저장 — 1차 범위 밖.
- YouTube 등 외부 스트리밍에서의 오디오 추출 — 구현하지 않는다.
- 전문가급 DAW 기능(자동 비트매칭, 고급 자동화 등) — 초기 버전에서는 단순/수동 우선.
